# Single Source of Truth в документации

Single Source of Truth (SSOT) в документации — это принцип, при котором у важной информации есть один основной источник, а все остальные страницы, инструкции и интерфейсные тексты ссылаются на него или переиспользуют его.

Цель SSOT не в том, чтобы хранить все в одном большом документе. Цель — сделать так, чтобы команда понимала, где находится актуальная версия факта, кто за нее отвечает и как изменение попадает во все зависимые материалы.

## Зачем это нужно

Без единого источника истины документация быстро расходится с продуктом:

- в API reference одно название поля, а в пользовательской инструкции другое;
- в базе знаний описан старый процесс, а в интерфейсе уже новый сценарий;
- support отвечает по устаревшей статье;
- QA проверяет кейс по требованиям, которые не совпадают с текущей реализацией;
- release notes не связаны с изменениями в API, сценариях и help center.

SSOT помогает снизить эти риски:

- уменьшает дублирование;
- упрощает обновление документации;
- помогает быстрее находить актуальную информацию;
- делает ответственность за данные видимой;
- снижает количество расхождений между задачами, макетами, API и пользовательскими материалами.

## Что может быть источником истины

Источник истины зависит от типа информации.

| Тип информации | Возможный source of truth | Что от него зависит |
|---|---|---|
| API-контракт | OpenAPI / Swagger-спецификация | API reference, SDK, примеры запросов, тесты |
| Пользовательский сценарий | Product spec, user story, BPMN, sequence diagram | Help center, QA-сценарии, onboarding, release notes |
| Интерфейсный текст | Figma, дизайн-система, UX-writing guideline | UI, email, push, help center |
| Терминология | Глоссарий, style guide, controlled vocabulary | Интерфейс, документация, support-ответы |
| Архитектура | C4, ADR, README сервиса, системная диаграмма | Техническая документация, runbook, onboarding команды |
| Операционные процедуры | Runbook, incident playbook | Support, DevOps, on-call, postmortem |
| Релизные изменения | Changelog, release notes, issue tracker | Документация версий, help center, уведомления пользователей |

## Подходы к SSOT

### 1. Docs as Code

Документация хранится в Git рядом с кодом или в отдельном репозитории. Изменения проходят review, собираются автоматически и публикуются через CI/CD.

Подходит для:

- API-документации;
- developer portals;
- технических гайдов;
- changelog и release notes;
- документации, которая должна обновляться вместе с кодом.

Инструменты: Git, GitHub / GitLab, Hugo, MkDocs, Docusaurus, Sphinx, Vale, Spectral, markdownlint.

### 2. API-first и contract-first

API-спецификация становится основным источником данных о методах, параметрах, схемах, ошибках и примерах. Документация, тесты и SDK строятся вокруг контракта.

Подходит для:

- REST API;
- публичных API;
- внутренних платформ;
- интеграций между сервисами.

Инструменты: OpenAPI, Swagger UI, Redoc, Stoplight, Spectral, Postman collections.

### 3. Component content и reuse

Повторяющиеся фрагменты хранятся отдельно и переиспользуются в разных документах: предупреждения, описания параметров, шаги настройки, юридические тексты, troubleshooting-блоки.

Подходит для:

- больших help center;
- enterprise-документации;
- документации с несколькими версиями продукта;
- материалов, которые публикуются в разные каналы.

Инструменты: DITA, AsciiDoc include, Sphinx includes, snippet-библиотеки, CCMS.

### 4. Глоссарий и терминологический контроль

Команда фиксирует термины, запрещенные варианты, правила перевода и контекст употребления. Это помогает не плодить разные названия одного и того же объекта.

Подходит для:

- сложных доменов;
- B2B-продуктов;
- продуктов с несколькими языками;
- команд, где документацию пишут разные люди.

Инструменты: глоссарий в Notion / Confluence, style guide, Vale, term checker, локализационные платформы.

### 5. Владелец и lifecycle документа

У каждой важной страницы есть owner, дата ревью и понятный процесс обновления. Это не технический, а организационный слой SSOT.

Подходит для:

- баз знаний;
- runbook;
- внутренних регламентов;
- документации поддержки.

Инструменты: Confluence, Notion, Jira / YouTrack, Linear, ownership-файлы, review reminders.

## Как выбрать подход

| Ситуация | Что выбрать |
|---|---|
| Документация зависит от кода | Docs as Code |
| Нужно описывать API без расхождений | OpenAPI как source of truth |
| Много повторяющихся фрагментов | Reuse: DITA, AsciiDoc includes, Sphinx includes |
| Есть путаница в терминах | Глоссарий и style guide |
| Документы устаревают без владельцев | Owner, review cadence, lifecycle |
| Нужно публиковать в несколько каналов | CCMS или structured authoring |

## Практический чеклист

Перед внедрением SSOT стоит ответить на вопросы:

- Где сейчас хранится актуальная версия факта?
- Кто может ее менять?
- Кто подтверждает изменение?
- Какие страницы зависят от этого факта?
- Как команда узнает, что зависимые документы нужно обновить?
- Можно ли переиспользовать фрагмент вместо копирования?
- Есть ли проверка в CI, ревью или регулярной ревизии?

Хороший SSOT — это не один идеальный инструмент, а связка процесса и структуры. Инструмент помогает хранить и публиковать данные, но источник истины работает только тогда, когда команда договорилась, где живет факт и как он обновляется.
