# Technical Writing / Documentation Engineering

Этот трек показывает, как я документирую архитектуру, API, интеграции и эксплуатационные сценарии.

## Что я закрываю как Technical Writer / DocOps

- Архитектурная документация: C4, sequence, границы системы, зависимости.
- API-документация: reference + quickstart + примеры ошибок.
- Интеграционные сценарии: контракты, retry, monitoring, troubleshooting.
- Документация данных: ERD, ключевые сущности, связи и ограничения.
- Docs as Code: структура, версионирование, review, поддерживаемость.

## Сквозной кейс: "Отклик на вакансию"

В работе готовлю отдельный набор артефактов под один сценарий:

- Architecture Overview
- API Reference
- Integration Guide
- Data Model (ERD)
- Event Contract (Kafka)
- Runbook

Цель: показать не отдельные заметки, а целостную инженерную документацию, пригодную для команды разработки и on-call.

## Как я работаю

1. Сначала фиксирую контекст: границы, участники, источники истины.
2. Затем описываю контракты: API, события, ошибки и ограничения.
3. После этого добавляю эксплуатационные секции: мониторинг, алерты, troubleshooting.
4. Поддерживаю все как docs-as-code: изменения проходят review вместе с кодом.
