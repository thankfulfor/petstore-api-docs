# Кейс: баннер о релизе

Кейс показывает, как писать предупреждение о технических работах так, чтобы пользователь понимал влияние на свои задачи.

## Задача

Нужно предупредить пользователей портала: завтра в 9:00 команда выкатывает релиз, и примерно два часа часть функций может работать нестабильно или быть недоступна.

## Что важно уточнить

Перед текстом баннера я бы задала команде вопросы:

- какие функции могут быть недоступны;
- можно ли продолжать работу в это время;
- есть ли риск потери данных;
- когда ожидается завершение работ;
- куда обращаться, если проблема сохранится после релиза.

## Плохой вариант

```text
Завтра релиз. Возможны ошибки.
```

Проблемы:

- нет времени начала;
- непонятно, сколько продлится нестабильность;
- не описано влияние на пользователя;
- нет рекомендации, что делать.

## Рабочий вариант

```text
29 апреля с 9:00 до 11:00 обновляем портал

В это время некоторые функции могут быть временно недоступны: создание заявок, загрузка документов и сохранение черновиков.

Если задача не срочная, вернитесь к ней после 11:00. Если ошибка сохранится после обновления, напишите в поддержку.
```

## Короткий вариант для компактного баннера

```text
29 апреля с 9:00 до 11:00 возможны перебои в работе портала.
Лучше не создавать заявки и не загружать документы до завершения обновления.
```

## Почему это лучше

| Элемент | Польза |
|---|---|
| Время | Пользователь понимает окно нестабильности |
| Затронутые функции | Можно оценить риск для своей задачи |
| Рекомендация | Понятно, что делать сейчас |
| Поддержка | Есть путь, если проблема не решилась |

## Связь с процессом документации

Такой баннер должен быть связан с release notes:

- баннер предупреждает до релиза;
- release notes объясняют, что изменилось после релиза;
- help center обновляется, если меняется пользовательский сценарий;
- support получает короткий script ответа на частые вопросы.
