# Release Notes

Главный принцип: release notes должны отвечать не “что команда сделала?”, а “что изменилось для читателя и нужно ли ему что-то предпринять?”.

## 1. Keep a Changelog

Подход группирует изменения по типам: что добавили, изменили, исправили, удалили или пометили как устаревшее. Такой формат удобен для API, SDK, библиотек и внутренних сервисов, где читателю важно быстро понять состав релиза.

**Структура:**

- `Added` — новые возможности.
- `Changed` — изменения в существующем поведении.
- `Fixed` — исправленные ошибки.
- `Deprecated` — то, что будет удалено позже.
- `Removed` — удаленная функциональность.
- `Security` — изменения по безопасности.

**Пример:**

```md
## 2.8.0

### Added
- Добавлен экспорт отчетов в CSV.

### Changed
- Обновлена логика расчета статуса задачи.

### Fixed
- Исправлен сброс фильтра после обновления страницы.
```

## 2. По аудиториям

Один релиз описывается отдельно для разных групп читателей: пользователей, администраторов, поддержки, разработки или QA. Подход помогает не смешивать пользовательскую пользу, операционные инструкции и технические детали.

**Структура:**

- Для пользователей — что изменится в интерфейсе и сценариях.
- Для поддержки — какие вопросы и ошибки ожидать.
- Для администраторов — какие настройки проверить.
- Для команды — какие сервисы, флаги, метрики и логи затронуты.

**Пример:**

```md
## Для пользователей
- Теперь можно закреплять важные задачи вверху списка.

## Для поддержки
- При жалобах на порядок задач проверять поле `pinned`.

## Для команды
- Feature flag: `task_pinning`.
```

## 3. Impact-first

Подход начинается не с перечня задач, а с эффекта для читателя. Он подходит для продуктовых релизов, где важно быстро показать ценность изменения.

**Структура:**

1. Что стало лучше.
2. Кого это затрагивает.
3. Какие изменения вошли в релиз.
4. Есть ли ограничения или действия со стороны пользователя.

**Пример:**

```md
## Быстрее работа с большими списками задач

Доски с 1000+ задачами теперь открываются быстрее, а фильтрация не блокирует интерфейс.

Что изменилось:
- ускорена первичная загрузка;
- уменьшено количество запросов к API;
- исправлено подвисание при смене фильтров.
```

## 4. Problem → Solution → Impact

Формат показывает логику изменения: какая была проблема, что сделали и какой результат получил пользователь. Он полезен для заметных улучшений, исправлений и изменений в сложных сценариях.

**Структура:**

- Проблема — что мешало пользователю или команде.
- Решение — что изменили в продукте или процессе.
- Результат — что теперь работает лучше.

**Пример:**

```md
## Улучшен поиск по задачам

Проблема: поиск не находил задачи по частичному совпадению в описании.

Решение: обновили индексирование и добавили поиск по описанию, тегам и исполнителю.

Результат: найти нужную задачу стало проще в больших проектах.
```

## 5. Breaking changes first

Если релиз может сломать интеграции, привычный сценарий или обратную совместимость, эти изменения выносятся в начало. Читатель должен сразу увидеть риск и способ миграции.

**Структура:**

1. Что несовместимо с прошлой версией.
2. Кого это затрагивает.
3. Что нужно изменить.
4. До какой даты нужно перейти на новый вариант.

**Пример:**

```md
## Breaking Changes
- Поле `user_id` теперь возвращается как строка, а не число.
- Endpoint `/v1/tasks/archive` удален.

## Migration
- Обновите парсинг `user_id`.
- Используйте `POST /v2/tasks/{id}/archive`.
```

## 6. Автогенерация из Conventional Commits

Release notes можно собирать из коммитов, если команда придерживается единого формата сообщений. Это снижает ручную работу, но итоговый текст все равно стоит редактировать: автогенератор хорошо собирает факты, но плохо объясняет пользовательский эффект.

**Пример входных данных:**

```txt
feat: add task templates
fix: prevent duplicate notifications
docs: update API examples
BREAKING CHANGE: remove v1 auth endpoint
```

**Что получается на выходе:**

```md
## Features
- Add task templates.

## Bug Fixes
- Prevent duplicate notifications.

## Documentation
- Update API examples.

## Breaking Changes
- Remove v1 auth endpoint.
```

## 7. User action required

Этот формат отдельно выделяет действия, которые должен выполнить пользователь, администратор или интегратор. Он особенно важен для API, enterprise-продуктов, безопасности и инфраструктурных изменений.

**Структура:**

- Что нужно сделать.
- Кому это нужно сделать.
- До какого срока.
- Что произойдет, если действие не выполнить.
- Где прочитать инструкцию.

**Пример:**

```md
## Требуются действия

До 2026-06-01 обновите webhook URL с `/v1/hooks` на `/v2/webhooks`.

Если этого не сделать, старые интеграции продолжат работать, но новые события не будут доставляться.
```

## Примеры релизных заметок

### Пример 1: Релиз профиля пользователя (v2.14.0)

**Дата:** 2026-05-01
**Статус:** Production

#### Что изменилось

- Добавили подтверждение смены email через код.
- Улучшили сообщения об ошибках при вводе кода.
- Добавили логирование причин неуспешной верификации.

#### Для пользователей

- Процесс смены почты стал понятнее: система сообщает, почему код не принят.
- При повторной отправке кода отображается ограничение по таймеру.

#### Для поддержки

- Новые коды ошибок: `EMAIL_CODE_EXPIRED`, `EMAIL_CODE_INVALID`, `EMAIL_RATE_LIMIT`.
- В тикетах запрашивать `requestId` для диагностики.

#### Для команды

- API: `POST /api/v1/profile/email/confirm`.
- Логи в Kibana: `profile-email-confirm`.
- Метрики: `email_confirm_success`, `email_confirm_failed`.

#### Риски и откат

- Риск: рост ошибок при старых клиентах.
- Откат: feature flag `email_confirm_v2` в значение `off`.

### Пример 2: Релиз уведомлений о занятиях (v2.15.0)

**Дата:** 2026-05-08
**Статус:** Production

#### Что изменилось

- Добавили новый тип уведомления: перенос занятия.
- Обновили шаблон push и email для родителей.
- Включили дедупликацию одинаковых событий в течение 10 минут.

#### Для пользователей

- Уведомления стали приходить без дублей при массовом переносе расписания.

#### Для поддержки

- Проверка дублей через `eventId` в логах.
- При жалобе на пропуск уведомления проверять `notification_status` и канал доставки.

#### Для команды

- Event: `lesson.rescheduled`.
- Consumer: `notification-service`.
- DLQ: `notification.events.dlq`.

#### Риски и откат

- Риск: неотправка в одном из каналов при таймауте провайдера.
- Откат: выключить обработчик `reschedule_notification_consumer`.
